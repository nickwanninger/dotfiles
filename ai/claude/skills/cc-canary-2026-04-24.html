4. Root-cause narrative

The biggest visible shift is not a clear quality break. It is a workload-and-output shift: tokens per user turn rise from 23,673 to 51,034, output tokens per API request rise from 255 to 643, and cache-read tokens per API request rise from 40,157 to 66,508. That combination says the post-split sessions are carrying more context and producing more text.

Read:Edit ratio does worsen in the headline table, and that is the strongest caution flag in the report. But it does not line up with the other “something is broken” indicators: interrupts fall 55%, explicit frustration falls to zero, and there are no reasoning-loop or self-admitted-error spikes. If there is a regression here, it is subtle enough that user correction behavior is not amplifying with it.

The thinking metrics point to instrumentation change more than capability collapse. The report marks the inflection as fallback_split_half at 0.0σ, there is no true pre-transition thinking baseline before 2026-03-25, and thinking visibility flips from absent to mostly visible to mostly redacted in a clean step-like pattern. That is consistent with a rollout or logging-policy change layered on top of normal task variation.

The cross-version section also cannot carry a regression claim. The apparent deltas are mostly claude-sonnet-4-6 versus unknown, not a stable old-versus-new comparison on the same task family. The most defensible root-cause read is therefore: heavier sessions and noisier attribution changed the aggregate metrics, while the user-facing correction signals stayed flat or improved.
